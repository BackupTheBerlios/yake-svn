Yake::Convention
Physical Organization  |  Logical Organization

Physical Organization

Namespaces

  1. Namespaces(similar to the directory structure) reflect the module structure. For instance, a class SmartPointer contained in module yake.core will be placed in a namespace yake::core.
  2. The class or code in general encapsulated by namespaces should not be displaced with tabs or spaces, this would consume too much horizontal space.

Naming conventions

  1. Namespaces are always lowercase. Example: 
    namespace yake {}
  2. In names which consist of more than one word, the words are written together and each word that follows the first is begun with an uppercase letter.
  3. Do not use identifiers which begin with one or two underscores, since these are reserved by the C++ environment (compiler, standard headers, ...).
  4. Do not use identifier names which differ only by the use of uppercase or lowercase letters.
  5. Use only generally accepted abbreviations.
  6. Variable names should suggest usage.
  7. The names of user-defined types start with an uppercase letter. Example: 
    class NasaSpaceShip;
    enum SpaceShipVendors { NASA, ESA };
  8. The names of interfaces (abstract classes) start with "I" and follow the same naming convention as other user-defined types do. Example: 
    class ISpaceShip;
  9. The names of functions start with an lowercase letter. Example: 
    void takeOff();
  10. The names of variables start with a lowercase letter. Example: 
    int numVertices;
  11. Pointer variables start with a lowercase p. Example: 
    int* pVertices;
  12. References start with a lowercase r. Example: 
    foo& rFoo;
  13. The names of member variables start with an additional letter m. Example: 
    int mNumVertices;
  14. The names of constants, enumeration values and #defines are uppercase-only with each word separated by an underscore. Example: 
    const int NUM_VERTICES = 100;
    In addition enumeration values may look like this example: 
    enum EngineState { kRunning, kOff };
    There's one exception. Error status codes in Yake look like this: 
    enum ErrorCode { eOk, eUnknown, eFileOpen };
  15. As an exception to the preceding rule, static member constants and enumerations in covariant classes like TypeTraits<> are to be written in the same way as variables are written to express their variability (for instance, the value of TypeTraits< T >::isPointer depends on the type T). Example: 
    static const bool isPointer = true;

Style - Classes

  1. The class should be splitted in sections identified with "//Class", "//Methods", "//Data" comments. Example: 
    class YAKE_CORE_API Log
{
DECLARE_CLASS( Yake::Core::Log )
// Types
public:
  enum Severity { INFORMATIONS, WARNINGS, ERRORS };

// Class
public:
  Log();
  virtual ~Log();

// Methods
public:
  void toLog( const String& rWhat, Severity severity, const String& rSource );

// Data
public:
  Signal2< void, Severity, String& > onLog;
private:
  YAKE_MAKE_SINGLETON( Log )
};
  2. Avoid inlining functions in the class definition.

Style - Functions

  1. When declaring functions, the entire function header (i.e. return type, function name, parameters and additional specifiers like inline and const) is to be written in one line.
  2. If there isn't enough space to put the entire function header into one line, put each parameter onto a new line. Example: 
    int addFourSheep(
    int sheepId,
    const Sheep& rSheep1,
    const Sheep& rSheep2,
    const Sheep& rSheep3,
    const Sheep& rSheep4 );
    Generally, though, you should try to avoid an excessive amount of function parameters.
     
  3. The left parenthsis is to be written directly after the function name. After the left parenthsis and in front of the right parenthsis, a space should be inserted. ( Same for conditions etc. ) Example: 
    int addSheep( int sheepId, const Sheep& sheepObject );

Style - Compound statements

  1. Braces ("{" ... "}") which enclose a block are to be placed in the same column, on separate lines directly before and after the block Exception: An empty body of a function that is inline implemented in the class declaration should be placed in one line (like in "{}").

Style - Flow control statements

  1. The flow control primitives if, else, while, for, do, try and catch should be followed by a block, even if the block is empty.
  2. When comparing a l-value with a r-value, always try to put the r-value on the left to avoid errors (when you forget a =). Example: 
    if( 0 == value )
  3. All empty blocks have to be either commented as such, or the two braces have to be on the same line. Example: 
    while( i-- > 0 )
{}
    and 
    while( i-- > 0 )
{
    // Empty loop.
}
    are both valid.
     

Style - Pointers and references

  1. The dereference operator ("*") and the address-of-operator ("&") should be directly connected with the type names in declarations and definitions in order to emphasize that they are part of the type definition. Example: 
    Sheep* pFavoriteSheep;
    Sheep& rBlackSheep;
  2. Declaration of multiple variables in one line is to be avoided, since it leads to mistakes especially in relation to pointers and references. Example: 
    // Do not do this!
int* pFirstInt, pSecondInt;
    In this example, pFirstInt is a pointer to int, whereas pSecondInt is an integer value.
